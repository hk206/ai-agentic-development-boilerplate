# ユーザーストーリー作成

`$ARGUMENTS` に基づいてユーザーストーリーを作成する。

## 手順

1. PRDまたは機能説明を読み込む（ファイルパスが指定された場合はそのファイルを読む）
2. ペルソナとユーザーフローを特定する
3. 各フローをユーザーストーリーに分解する
4. 受け入れ基準を定義する
5. 優先度とポイント見積もりを付与する
6. `docs/user-stories/` に保存する

## ストーリー形式

テンプレート: [templates/user-story-template.md](templates/user-story-template.md)

各ストーリーは以下の形式で記述する:

```markdown
### US-[番号]: [ストーリータイトル]

**ストーリー**: [ペルソナ]として、[何をしたい]、なぜなら[理由]だから。

**受け入れ基準**:
- [ ] 基準1
- [ ] 基準2
- [ ] 基準3

**見積もりポイント**: [1 / 2 / 3 / 5 / 8 / 13]
**優先度**: [Must / Should / Could / Won't]
**関連PRD**: [PRDファイルへの参照]
```

## 分解のガイドライン

- 1ストーリー = 1〜3日で完了できる粒度にする
- 技術タスクではなくユーザー価値の単位で分割する
- 依存関係がある場合は明記する
- 非機能要件もストーリー化する（パフォーマンス、セキュリティ等）

## 優先度の判断基準

- **Must**: ローンチに必須。これがないとプロダクトとして成立しない
- **Should**: 重要だが、初回リリースに間に合わなくても許容可能
- **Could**: あれば良いが、なくても問題ない
- **Won't**: 今回のスコープ外（将来の検討事項）
