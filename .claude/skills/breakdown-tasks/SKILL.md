# タスク分解

`$ARGUMENTS` に基づいて実装タスクを分解する。

## 手順

1. ユーザーストーリーまたはPRDを読み込む
2. 各ストーリーを実装タスクに分解する
3. タスク間の依存関係を整理する
4. 見積もりと優先度を付与する
5. `docs/tasks/` に保存する

## 分解の基準

- 1タスク = 1PR（半日〜1日で完了できる粒度）
- 依存関係を明記する
- 技術タスクとビジネスタスクを区別する
- テスト作成タスクを含める

## タスク種別

- `feature`: 機能実装
- `test`: テスト作成
- `design`: UI/UX設計
- `infra`: インフラ・設定
- `docs`: ドキュメント作成
- `refactor`: リファクタリング

## 出力形式

### タスク一覧

| # | タスク名 | 種別 | 依存 | 見積もり | 優先度 | ストーリー |
|---|---------|------|------|---------|--------|-----------|
| TASK-001 | | feature | - | 0.5d | High | US-001 |

### 各タスクの詳細

```markdown
#### TASK-[番号]: [タスク名]

- **種別**: feature / test / design / infra / docs / refactor
- **説明**: 実装内容の具体的な説明
- **受け入れ基準**: 完了条件
- **技術的考慮事項**: 実装上の注意点
- **関連ファイル**: 変更が必要なファイル
- **依存タスク**: 先に完了が必要なタスク
- **見積もり**: 0.5d / 1d / 2d
- **関連ストーリー**: US-XXX
```

## タスク管理ツールへの登録

タスク分解完了後、ユーザーに登録先を確認すること:

- **Notion**: `/sync-notion` スキルを使用してNotionのスプリントデータベースに登録する
- **GitHub Issues**: `gh issue create` コマンドで Issue として作成する
