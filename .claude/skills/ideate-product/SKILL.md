# プロダクト企画・構想

`$ARGUMENTS` に基づいてプロダクト/サービスの企画を行う。

## このスキルがカバーする範囲

「何を作るべきか？」を明確にする上流工程全体:

1. 課題の発見・定義
2. 市場・競合分析
3. プロダクトビジョン・戦略
4. ビジネスモデル設計
5. ユーザーリサーチ計画
6. MVP仮説の策定

## 手順

### Phase 1: 課題の探索

ユーザーとの対話を通じて以下を明確にする:

- **誰の**課題を解決するのか（ターゲット）
- **どんな**課題・ペインポイントがあるのか
- **なぜ今**その課題を解決する必要があるのか
- 既存の解決手段は何か、それらの不満は何か

不明点は必ずユーザーに質問する。仮定で進めない。

### Phase 2: 市場・競合分析

WebSearchを活用して調査する:

- **市場規模**: TAM / SAM / SOM の推定
- **市場トレンド**: 成長率、注目される動き
- **競合分析**: 直接競合・間接競合のマッピング
- **ポジショニング**: 差別化ポイントの特定

#### 競合分析マトリクス

| 競合 | 強み | 弱み | 価格帯 | ターゲット | 差別化ポイント |
|------|------|------|--------|-----------|---------------|
| | | | | | |

### Phase 3: プロダクトビジョン

以下のフレームワークで整理する:

#### プロダクトビジョンステートメント

```
[ターゲットユーザー] にとっての
[課題/ニーズ] を解決する
[プロダクトカテゴリ] である。
[競合/代替手段] とは異なり、
[主要な差別化ポイント] を提供する。
```

#### バリュープロポジション

- **顧客の仕事（Jobs to be Done）**: ユーザーが達成したいこと
- **ペイン**: 現状の課題・不満
- **ゲイン**: 理想の状態
- **ペインリリーバー**: 課題をどう解決するか
- **ゲインクリエイター**: どんな価値を提供するか

### Phase 4: ビジネスモデル

Lean Canvasで整理する（テンプレート: [templates/lean-canvas-template.md](templates/lean-canvas-template.md)）:

| 項目 | 内容 |
|------|------|
| **課題** | 上位3つの課題 |
| **顧客セグメント** | ターゲットユーザー |
| **独自の価値提案** | 明確で説得力のあるメッセージ |
| **解決策** | 課題に対する解決策 |
| **チャネル** | ユーザーへの到達経路 |
| **収益の流れ** | マネタイズモデル |
| **コスト構造** | 主要コスト |
| **主要指標** | 計測すべきKPI |
| **圧倒的優位性** | 簡単にコピーできない強み |

### Phase 5: ユーザーリサーチ計画

- **ペルソナ仮説**: 想定ユーザー像の詳細定義
- **検証すべき仮説**: リストアップ
- **リサーチ手法**: インタビュー / アンケート / プロトタイプテスト
- **質問リスト**: インタビューで聞くべき質問

### Phase 6: MVP仮説

- **MVPの定義**: 最小限で価値を検証できるプロダクト像
- **含める機能**: Must Haveのみ
- **含めない機能**: 意図的に除外するものとその理由
- **成功基準**: MVPで検証する仮説と成功の定義
- **次のステップ**: MVP後のロードマップ概要

## 出力

以下のドキュメントを `docs/product/` に保存する:

- `[product-name]-vision.md` — プロダクトビジョン・戦略
- `[product-name]-market-analysis.md` — 市場・競合分析（調査量が多い場合）
- `[product-name]-lean-canvas.md` — ビジネスモデル（独立させる場合）

小規模なプロダクトの場合は1つのファイルにまとめてよい。

## 次のアクション

企画が固まったら以下のスキルに引き継ぐ:

1. `/create-prd` — 機能単位のPRD作成
2. `/create-user-stories` — ユーザーストーリー分解
3. `/breakdown-tasks` — 実装タスク分解

## 注意事項

- ユーザーとの対話を重視する。一方的に大量のドキュメントを生成しない
- 各Phaseの完了後にユーザーのフィードバックを求める
- 仮定は `[仮説]` マークで明示し、検証が必要であることを示す
- 完璧を目指さない。イテレーションで改善する前提で進める
